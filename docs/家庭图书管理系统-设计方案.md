## 0. 文档导读

本设计方案描述一个**面向家庭的图书管理系统**，核心特征：

- **以 Agent Skills + CLI 为大脑**：所有业务能力封装为「技能（Skill）」与可被命令行调用的工具（CLI），由 AI Agent 编排。
- **后端运行在家庭服务器上**：常驻服务 + SQLite 数据库，局域网内可访问，数据完全私有。
- **前端是 IM 通信工具**：用户通过飞书 / 微信 / Telegram 发消息（文字、图片、扫码、自然语言）即可完成入库、记录、查询。
- **强图像识别能力**：扫书封、拍书脊、识别 ISBN 条码，自动补全书籍元数据。
- **数据可扩展**：不仅存文字，还能挂载图片、链接、Markdown 文档作为任意条目的扩展内容。

分三期迭代，本文档重点设计 **第一期（原型/链路打通）** 与 **第二期（数字书架/可视化）**，第三期（家庭间交换）仅做接口与架构层面的预留。

## 1. 项目目标与范围

### 1.1 总体愿景

把"家里有哪些书、谁在读、读到哪、花了多少钱买的、读后有什么笔记"这些信息，用最低的录入成本沉淀到一个**私有、可检索、可扩展、可视化**的系统中，并最终让**实体书与电子书在同一套逻辑下统一管理**。

### 1.2 核心设计原则

| 原则 | 说明 |
|---|---|
| 录入成本最低 | 一张照片 / 一个 ISBN / 一句话就能入库，重活交给 Agent 和后端自动补全 |
| 数据私有 | 数据全部落在家庭服务器的 SQLite，不依赖云端 SaaS |
| 能力即技能 | 每个业务动作是一个可复用 Skill，Agent 负责"理解意图→选技能→调 CLI→落库" |
| 模式可扩展 | 数据库表结构支持新增字段、附件、自定义属性，不因需求变化频繁改表 |
| 渐进式交付 | 第一期单机跑通链路，第二期加可视化，第三期再谈互联 |

### 1.3 范围边界

- ✅ 范围内：个人/家庭藏书管理、阅读进度、购买记录、读书笔记、数字书架、局域网浏览与在线阅读。
- 🔶 预留：家庭之间的图书交换、信息发布与流转（第三期，本期仅留接口）。
- ❌ 暂不做：电商/二手交易闭环、社交评论体系、跨家庭权限与隐私合规体系。

### 2.2 关键说明：Skill 与 CLI 的关系

- **Skill（SKILL.md）** = 给 Agent 看的"说明书"：什么时候用、需要哪些信息、调用哪个 CLI 命令、如何回话。
- **CLI（`bookshelf` 命令）** = 给程序执行的"动作"：参数化、可脚本化、稳定输出 JSON，便于 Agent 解析。
- **后端 API** = 真正的业务实现与数据落地。

> 这样分层的好处：Agent 可以灵活组合技能应对自然语言，而真正的数据操作走稳定的 CLI/API，避免 Agent 直接写 SQL 带来的不可控。

### 2.3 一次典型交互（拍照入库）

```
用户在飞书发一张书封照片，附文字"这本买了，38块"
   │
   ▼ Channel Adapter 下载图片 + 文本，封装 UnifiedMessage
   ▼ Agent 多模态识别 → 提取书名/作者，识别意图=入库+购买记录
   ▼ Agent 调用 Skill「book-intake」
   ▼ Skill 指示执行：bookshelf add --image cover.jpg --price 38
   ▼ CLI → POST /api/v1/books/intake（带图片）
   ▼ 后端：识别ISBN/书封 → 查外部书目源补全元数据 → 落库 books + purchase_records
   ▼ 返回结构化结果 JSON
   ▼ Agent 组织自然语言回复："已入库《xxx》(作者)，购买价 ¥38，已记录到你的书架"
```

| 层 | 选型 | 理由 |
|---|---|---|

| 图像/书封识别 | Agent 多模态（首选）+ 可选 PaddleOCR 兜底 | 优先用 Agent 视觉能力读封面文字，OCR 作为离线兜底 |
| 书目元数据源 | OpenLibrary / Google Books / 国家图书馆 / 豆瓣镜像 | 多源补全，ISBN→书名作者出版社等 |
| 任务/常驻 | systemd 或 docker-compose | 家庭服务器常驻、开机自启、崩溃重启 |
| 前端可视化(二期) | 后端 SSR(Jinja2) 或轻量 SPA(Vue/React) | 局域网数字书架浏览/在线阅读 |
| 电子书阅读(二期) | epub.js / pdf.js | 浏览器内阅读 EPUB/PDF |

## 4. 数据库设计（核心，重点强调可扩展性）

### 4.1 设计思路

为兼顾"结构化"与"可扩展"，采用 **核心表（强结构）+ 通用附件表 + EAV 自定义字段表 + JSON 扩展列** 三种机制组合：

1. **核心表**：高频、稳定的字段直接建列（书、副本、购买、进度、笔记、成员）。
2. **通用附件表 `attachments`**：任何实体都能挂载 图片 / 链接 / Markdown / 文件，解决"不仅存文字"的需求。
3. **自定义字段表 `custom_fields`（EAV）**：用户临时想记的非标准属性（如"签名本""绝版""借给了谁"）无需改表即可记录。
4. **JSON 扩展列 `extra`**：每张核心表预留一个 JSON 列，存半结构化补充信息。

> 三者配合，使得"未来新增需求"绝大多数情况下**不需要做破坏性表结构变更**。

### 4.2 ER 关系概览

```
members ──< reading_progress >── books ──< book_copies
   │                                │
   └──< reading_notes >─────────────┤
                                    ├──< purchase_records
                                    ├──< book_tags >── tags
                                    └──< attachments（多态：可挂任意实体）
                                    └──< custom_fields（多态：EAV 扩展）
```

### 4.4 索引与约束建议

- `books.isbn13` 唯一索引；`books.title` 普通索引（模糊查询）。
- `reading_progress(book_id, member_id)` 唯一约束。
- `attachments(entity_type, entity_id)` 联合索引。
- `custom_fields(entity_type, entity_id)` 联合索引。
- SQLite 开启 `PRAGMA journal_mode=WAL;` 提升并发读性能；`PRAGMA foreign_keys=ON;` 保证外键约束。

### 4.5 文件存储约定

数据库只存**相对路径**，真实文件放文件系统，便于备份与迁移：

```
data/
├── bookshelf.db                 # SQLite 主库
├── covers/                      # 封面图  {isbn or uuid}.jpg
├── attachments/                 # 附件（图片/文件）
│   └── {entity_type}/{entity_id}/...
├── ebooks/                      # 电子书原文件（二期在线阅读）
│   └── {book_id}/book.epub
└── notes/                       # 独立 MD 笔记文件（可选）
```

## 5. Skills + CLI 设计（能力层）

### 5.1 CLI：`bookshelf`

统一命令行入口，所有子命令默认 `--json` 输出结构化结果，便于 Agent 解析。

```bash
# 入库（核心）：支持 ISBN / 图片 / 纯文字
bookshelf add --isbn 9787121234567
bookshelf add --image /path/cover.jpg
bookshelf add --title "活着" --author "余华"
bookshelf add --image cover.jpg --price 38 --channel 当当   # 入库同时记购买

# 查询
bookshelf find --keyword "三体"
bookshelf find --author "刘慈欣" --status reading
bookshelf show --id 12                      # 查看一本书全部信息（含附件/进度/购买）
bookshelf list --shelf "客厅书架A"          # 按位置列书

# 阅读进度
bookshelf progress --book-id 12 --member 张三 --page 120
bookshelf progress --book-id 12 --status finished --rating 5

# 购买记录
bookshelf purchase --book-id 12 --price 45 --channel 京东 --order-no XXXX

# 笔记（Markdown）
bookshelf note --book-id 12 --type excerpt --content "## 摘录\n人是为活着本身而活着"

# 附件（图片/链接/MD）
bookshelf attach --entity book --id 12 --type link --url https://book.douban.com/xxx
bookshelf attach --entity book --id 12 --type image --file /path/扉页签名.jpg

# 自定义字段
bookshelf field --entity book --id 12 --key 签名本 --value 是

# 报表/统计（二期可视化数据来源）
bookshelf stats --by category
bookshelf stats --spending --year 2026
```

输出示例（`--json`）：

```json
{
  "ok": true,
  "action": "add",
  "book": { "id": 12, "title": "活着", "authors": ["余华"], "isbn13": "9787506365437" },
  "matched_source": "openlibrary",
  "message": "已入库《活着》"
}
```

### 5.2 Skills 清单（SKILL.md）

每个技能一个目录 + 一份 SKILL.md，描述触发场景、所需信息、调用的 CLI、回话风格：

| Skill | 触发场景 | 主要动作 |
|---|---|---|
| `book-intake` | 用户拍书封/发ISBN/说"我买了本书" | 识别→补全元数据→`bookshelf add`（可带购买） |
| `reading-tracker` | "我读到第X页/读完了/在读" | `bookshelf progress` |
| `purchase-logger` | "这本花了X元，在Y买的" | `bookshelf purchase` |
| `book-query` | "我有没有XX这本书/我家关于历史的书" | `bookshelf find` / `show` |
| `note-taker` | "记一段摘录/写点感想" | `bookshelf note` |
| `shelf-report` | "我家有多少书/今年买书花了多少" | `bookshelf stats` |

SKILL.md 模板要点：
- **触发条件**：什么样的用户输入应启用本技能。
- **信息收集**：缺少关键信息（如同名书需确认是哪一本）时如何向用户追问。
- **执行步骤**：调用哪个 CLI、参数怎么填。
- **回复规范**：用自然语言确认结果，附上识别到的关键信息让用户校验。
- **异常处理**：识别失败、外部源查不到、重复入库时的兜底话术。

### 6.1 入库流程（`/books/intake` 内部逻辑）

```
接收 (image? / isbn? / title+author? / price?…)
  │
  ├─ 若有 image：
  │     1) 条码识别 pyzbar → 拿到 ISBN-13
  │     2) 无条码 → 多模态/OCR 读封面文字 → 提取书名作者
  │
  ├─ 若有 ISBN：去外部书目源补全（OpenLibrary→Google→…多源回退）
  │
  ├─ 去重：按 isbn13 / (title+author) 查是否已存在
  │     已存在 → 提示"已在书架"，可选择加副本/加购买记录
  │
  ├─ 落库 books（+ 下载封面到 covers/）
  ├─ 若带 price → 落库 purchase_records
  ├─ 默认创建一个 physical 副本（可配置）
  └─ 返回结构化结果
```

### 6.2 图像识别能力（重点）

分三级，优先级从高到低：

1. **ISBN 条码识别**（最准）：`pyzbar` 解析书封/封底 EAN-13 条码 → ISBN → 外部源精确补全。
2. **封面多模态理解**（首选无条码场景）：交给 Agent 的视觉能力读取书名/作者/出版社关键文字，再用文字去外部源检索匹配。
3. **OCR 兜底**（离线/无 Agent 视觉时）：`PaddleOCR` 提取封面文字，规则+检索匹配。

> 识别结果一律**回传给用户确认**（"识别到《XXX》，作者 YYY，对吗？"），避免脏数据。

### 6.3 外部书目元数据源（多源回退）

| 源 | 用途 | 备注 |
|---|---|---|
| OpenLibrary API | ISBN→元数据、封面 | 免费、开放、英文为主 |
| Google Books API | ISBN/关键词检索 | 中英文较全，需 API Key |
| 国家图书馆 / ISBN 中心 | 中文书兜底 | 抓取/接口视可用性 |
| 豆瓣（镜像/第三方） | 中文书评分简介 | 官方 API 已关停，谨慎使用 |

抽象成 `MetadataProvider` 接口，逐个尝试直到补全成功，便于增删源。

### 7.1 统一消息模型

```python
UnifiedMessage = {

  "raw": {...}

### 7.2 各渠道接入方式

| 渠道 | 接入方式 | 一期是否做 |
|---|---|---|
| 飞书 | 企业自建应用 + 事件订阅(Webhook) + 机器人 | ✅ 首选 |
| Telegram | Bot API（长轮询或 Webhook） | 🔶 二期可选 |
| 微信 | 个人微信(itchat 类，不稳定/有风险) 或 企业微信(推荐) | 🔶 二期，优先企业微信 |

> 一期只接 **飞书** 一个渠道，把"消息→Agent→CLI→后端→回复"链路打通即可。Channel Adapter 设计为可插拔，后续加渠道只实现一个适配器。

### 7.3 安全与鉴权

- 仅允许 `members.channel_bindings` 中登记的账号触发操作（白名单）。
- 后端服务仅监听局域网，IM Webhook 通过家庭服务器的反向代理(或内网穿透)接入。
- 媒体文件下载后本地存储，不外传。

## 8. 三期迭代计划

### 第一期：原型 / 链路打通（单机版）🎯 本期重点

**目标**：把"录入→落库→查询"的核心链路完整跑通，打磨"图片/文字/扫码三种方式快速入库"的单点体验。

**交付物**：
1. SQLite 数据库 + **12 张核心表**（见 Schema 对照文档 §3 表清单）+ Alembic 迁移。
2. FastAPI 后端：§6 端点清单（health / books CRUD / intake / progress / reading-logs / purchases / notes / attachments / custom-fields / stats / recognize / members）。
3. 图像识别：ISBN 条码识别 + 封面识别（条码 + 书名检索；Agent 多模态/OCR 兜底 🔶 二期）+ 外部源补全（OpenLibrary + Google Books + 国图）。
4. `bookshelf` CLI：`add / find / show / progress / reading-log / purchase / note / stats / recognize / health`。
5. Skills：`book-intake`、`book-query`、`reading-tracker`、`purchase-logger`、`note-taker`、`shelf-report`。
6. ~~飞书 Channel Adapter~~ → Skills 接入 OpenClaw / Hermes 等 Agent（不单独开发 channels/）。
7. 部署：家庭服务器 systemd/docker 常驻 + 数据备份脚本。

**验收标准（一期 Done 的定义）**：
- 发一张书封照片，10 秒内收到"已入库《书名》"并能在数据库查到。
- 发 ISBN 文本能入库。
- 发"我有没有 XX"能查到/查不到的明确回复。
- 发"读到 100 页"能更新进度。
- 数据可通过 CLI 与飞书两种方式查询一致。

**里程碑拆分（建议顺序）**：
| 步骤 | 内容 |
|---|---|
| M1 | 建库 + ORM 模型 + Alembic + 基础 CRUD API |
| M2 | CLI `add/find/show` 跑通（先用纯文字/ISBN） |
| M3 | 图像识别（条码→ISBN→补全；封面识别） |
| M4 | Skills 编写 + Agent 编排（本地命令行模拟对话） |
| M5 | 飞书接入，端到端跑通 |
| M6 | 部署常驻 + 备份 + 文档 |

### 第二期：数字书架 / 可视化 / 统一管理

**目标**：让藏书"看得见"，实体书与电子书统一管理，局域网可浏览、可在线阅读。

**交付物**：
1. **Web 数字书架**（局域网访问）：
   - 书架总览（按分类/位置/状态分组的网格视图，封面墙）。
   - 书籍详情页（元数据 + 副本 + 进度 + 购买 + 笔记 + 附件）。
   - 搜索/筛选/标签。
2. **可视化生成**：
   - "我的书架概览图"（封面墙拼图、藏书统计图：按分类/年份/花费）。
   - 阅读统计（在读/已读/年度阅读量、花费趋势）。
   - 可导出图片分享。
3. **实体书 + 电子书统一**：
   - `book_copies` 区分实体/电子；电子书上传 EPUB/PDF。
   - 在线阅读（epub.js / pdf.js），阅读进度与 `reading_progress` 打通。
4. **更多渠道**：Telegram / 企业微信接入。

**验收标准**：
- 局域网打开网页能看到全家藏书封面墙并筛选。
- 能一键生成"我家书架概览图"。
- 上传一本 EPUB 能在浏览器里翻页阅读，进度回写数据库。

### 第三期：家庭间交换（仅架构预留，本期不实现）

**目标（远期）**：不同家庭之间的图书交换、借阅、信息发布与流转。

**本期只做预留**：
- 数据模型为"书可被借出/转移"预留 `book_copies.status=lent_out` 与 `extra` 借阅信息。
- API 与 ID 体系预留"跨家庭实体标识"扩展位。
- 不投入实现，避免过早设计互联/隐私/信任体系带来的复杂度。

## 10. 部署与运维

- **常驻**：`docker-compose`（推荐，环境隔离）或 `systemd` 服务，开机自启、崩溃重启。

- **备份**：每日 `sqlite3 .backup` + `data/` 目录打包，保留最近 N 份；可同步到 NAS。
- **可观测**：`operation_logs` 记录关键操作；后端结构化日志，便于排查识别失败等问题。

## 11. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 中文书目数据源不稳定（豆瓣关停等） | 元数据补全不全 | 多源回退 + 允许手工补全 + 缓存已查结果 |
| 封面识别准确率 | 入脏数据 | 识别结果强制用户确认；条码优先 |
| 个人微信接入风险 | 封号/不稳定 | 优先企业微信/飞书，个人微信不作为主渠道 |
| IM webhook 暴露公网 | 安全风险 | 白名单 + 签名校验 + 内网穿透最小化暴露 |
| SQLite 并发写 | 家庭量级基本无忧 | WAL 模式；未来量大可迁 PostgreSQL（ORM 已抽象） |
| 需求频繁变动 | 频繁改表 | attachments + custom_fields + JSON extra 三机制吸收变化 |
